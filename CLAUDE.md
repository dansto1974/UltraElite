# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A browser-based Elite clone ("Vector Elite"), currently implemented entirely in a single file, `index.html` (~2,600 lines: CSS, then one `<script>` block). No build step, no package manager, no dependencies — it's an HTML file you open or serve directly. There is no test suite and no linter configured.

This is a for-fun vibe-coding experiment: the goal is a modern-spin take on 1984 Elite, built almost entirely through conversational iteration rather than a planned spec. Priorities are "does it feel good to fly/fight/trade" and "is it fun to keep bolting features on," not production hygiene. Keeping everything in one `index.html` has been a nice constraint so far (paste-and-run, no tooling), but it's not sacred — separate asset files (sprites, textures, sound, even a split-out JS module or two) are fine if a feature genuinely wants them. Don't over-engineer or add process (build steps, test harnesses, frameworks) unless it's clearly earning its keep.

## Commands

- **Run it**: open `index.html` directly in a browser, or serve it locally (`npx serve .` or `python3 -m http.server`) — a static server avoids any `file://` restrictions.
- **No build/lint/test commands exist.** Verify changes by loading the page in a browser and checking the console; there's nothing to compile.

## Rendering-space rule

Most visual regressions in this project have come from drawing world-attached things with screen-space canvas primitives. Treat this as a hard rule:

- Anything attached to a ship, station, planet, ring, or surface must be represented as local/world 3D points or directions, transformed with `cameraTransform`/`worldDirToCam`, and then passed through `project()`.
- Raw screen-space `ctx.ellipse()`, `ctx.arc()`, gradients, and billboards are fine for UI, transition tunnels, particles, explosions, glows, targeting marks, and other effects that are deliberately camera-facing.
- Planet surface features must use the helpers near `drawPlanetSurface` (`surfaceEllipseDirs`, `projectedSurfacePatch`, `drawProjectedCloud`, etc.). Do not add clouds, craters, city patches, gas bands, caps, or shadows as 2D ellipses/gradients on the already-projected planet disc; they will rotate with ship roll.
- Ship/station hull details should go through model geometry, `collectSolidFaces`, `collectSolidDetails`, or fixed object-local UVs. Do not derive texture/detail axes from camera-space normals per frame.

## Architecture

Everything lives in `index.html`, in this rough top-to-bottom order:

1. **CSS** — CRT/vector-scope aesthetic (`:root` custom properties for the green/amber/cyan palette). Layout is CSS Grid: `#app` splits into the flight view + side panel (`.main`), the dashboard strip, and the message/command footer. Two `@media` breakpoints (860px, 520px) collapse the layout for mobile.

2. **Math/camera layer** (`vec`, `quat*` functions, `rotatePoint`, `makeCamera`, `cameraTransform`, `forwardVector`) — a quaternion-based camera. The player's ship never moves; instead `game.camera` (position + orientation quaternion) moves through a static `game.objects` list, and world positions are transformed into camera space for projection. `project()` does the perspective divide into screen space.

3. **Ship blueprints** (`MODELS`, `buildBlueprint`) — each ship is defined as `verts`/`edges`/`edgeFaces`/`normals` (+ optional `details` for windows/engines/trim). `buildBlueprint()` derives polygon `faces` from the edge/face-adjacency data at load time, ordering vertices by angle around each face's centroid and correcting winding via `faceNormal`. If you add a new ship, you only need to supply verts/edges/edgeFaces/normals — faces are computed, not authored by hand. Any ship can also be overridden at runtime by a hand-edited `.obj` file in `ship_models/` — see the note under "Notable systems" below. This is the one deliberate exception to "everything lives in index.html": real geometry editing needs a real 3D tool, not more JS literals.

4. **Galaxy generation** (`GalaxyFactory`, `galaxies`) — deterministic, seeded system generation modeled on the original Elite's algorithm: three 16-bit seed words are "twisted" per system to derive economy/government/tech/population/name/species. All 8 galaxies for the session are generated up front at module load (`Array.from({ length: 8 }, ...)`). `LAVE_INDEX`/`DISO_INDEX` are found by name rather than hardcoded indices.

5. **Game state** — a single mutable `game` object holds all mutable state (position, credits, cargo, equipment, current/target system, camera, `objects` array of in-flight entities, etc). There's no state management abstraction; functions read/write `game.*` directly.

6. **Simulation** (`update`, `updateShipAI`, `steerVelocity`, `updateStation`, `updateAutoDock`) — called once per animation frame with `dt`. NPC ships pick an `aiMode` (`cruise` / `orbit` / `formation` / `attack`) and `steerVelocity` moves them toward a per-mode target position. Combat, docking legality, fuel/shield/energy regen, and legal-status decay all happen here.

7. **Rendering** (`drawSpace`, `drawWireModel`, `drawStation`, `collectSolidFaces`/`collectSolidDetails`, `drawSolidItems`) — two graphics modes selected by `game.graphicsMode`: `"wire"` draws model edges directly; `"solid"` back-face-culls and flat-shades polygons against a light direction derived from the sun's position (`starLightDirection`), then z-sorts and fills them. The scanner (`drawScanner`) is a separate small canvas driven by the same `game.objects`.

8. **UI panels** (`renderPanel`, `renderMarket`/`renderChart`/`renderEquip`/`renderStatus`/`renderShip`/`renderHelp`/`renderCheats`, `attachPanelEvents`) — the side panel is plain string-templated HTML re-rendered into `panelBody.innerHTML` on every state change; `attachPanelEvents()` re-binds click handlers after each render since the DOM is fully replaced. There's no virtual DOM / diffing — this is intentional and fine at this scale, just be aware a full panel re-render happens on most actions.

9. **Input** — a single `keydown`/`keyup` listener maintains a `keys` Set for held movement keys (WASD/QE/+-), plus one-shot key actions (space to fire, J/O/C/M/R/G/1-4) handled inline in the `keydown` handler. Footer buttons call the same functions as their keyboard shortcuts.

10. **Persistence** — `commanderSaveData`/`applyCommanderSave` own the save schema used by local browser saves and downloadable commander files. `saveGame`/`loadGame` still use `localStorage` under `vectorEliteSave`; the Status panel also has Download/Upload Commander controls for `.json` files. `applyCommanderSave` has one hardcoded migration check for saves predating `galaxySeedVersion` (remaps `current`/`target` from raw indices 0/1 to the found Lave/Diso indices).

## Notable systems (ported from / inspired by the BBC Master source)

- **Hull texturing** (`buildHullTexture`, `getHullTexture`, `drawTexturedTriangle`, `drawFaceTexture`) — solid-shaded faces get a real texture-mapped brushed-metal pass, not just a flat lit fill. UV coordinates are computed **once, from static object-local vertex positions** — `buildBlueprint` computes `model.faceUV` alongside `model.faces` (reusing the same per-face `u`/`v` basis already used for winding order); the station uses the exact same `buildBlueprint` pipeline (see below) so it gets this for free too. This matters: UV must **not** be derived from the rotating camera-space normal per frame — that basis isn't rigidly attached to the object (picture combing a hairy ball) and makes the texture swim/flip independently of the actual rotation instead of staying glued to the hull. `drawTexturedTriangle` is the classic canvas affine-transform trick: fan-triangulate the face, solve a 2x2 linear system per triangle to map the fixed local UV space onto current screen space, clip, `drawImage`. Composited with `multiply` blending over the existing flat-shaded fill so per-face lighting is preserved. Culled below a screen-space area threshold for performance. **Perspective correction** (`drawUVLayerPersp`/`subdivideTexTri`, gated on `item.project`+`item.camVerts` being supplied): the plain affine trick is linear across a triangle and can't represent perspective foreshortening, so on a *large near* face (the station up close, or a big ship like the Anaconda) the linear stretch visibly shifts as the face rotates — the texture appears to "swim" / stay locked to the viewer's turn even though the UV is correctly fixed. Fixed by recursively subdividing each fan triangle **in camera space** (splitting the longest screen edge at its 3D midpoint, re-projecting) until every sub-triangle is under ~48px, so affine error per triangle is negligible; UV interpolates linearly right alongside the camera-space midpoints because object→camera is rigid (UV affine in object space ⇒ affine in camera space). Small ship faces never exceed the threshold, so they cost/look exactly as before. `collectSolidFaces` now attaches `camVerts` + a bound `project` closure to each metal face item, supplied by `drawWireModel`/`drawStation`; callers without a projector (ship-diagram preview) fall back to the flat affine path. **This is the real cause of any residual "texture linked to ship rotation" report — it is not the old rotating-UV-basis bug (that stays fixed); it is affine perspective error, worst on the physically-large station.**
- **Station model** (`STATION_MODEL`, `STATION_SIZE`) — the Coriolis station's octagonal-prism hull is a `buildBlueprint()` model like every ship, not bespoke inline verts/edges/faces. Since its 10 faces (2 octagon caps + 8 side quads) are simple, `edgeFaces` is hand-authored directly (mapping which of the 10 faces each of the 24 edges borders) rather than relying on any derivation — `buildBlueprint` still does the winding/UV work from that adjacency data. Verts are baked to real hull scale (`* STATION_SIZE`) up front so the shared UV/texture pipeline needs no station-specific scale handling. `drawStation` stays a separate function from `drawWireModel` (not just reuse) because the station's rotation convention is single-axis roll (`o.rot` passed as `roll`), not the yaw/pitch/roll a ship uses, and it has the extra docking-slot-marker overlay ships don't.
- **AI roles & flee behavior** (`updateShipAI`, `fireAtPlayer`) — ships carry an explicit `role` (`"trader"|"pirate"|"police"`) and `maxHp` set at spawn. Traders that get spooked flee rather than fight; hostile non-police ships below ~32% hp break off and flee off-screen. This replaces the old behavior where a startled trader would flip straight into an attacker.
- **ECM and missiles** — hostile ships now sometimes fire a missile instead of a laser ping; `game.equipment.ecm` gives a real (75%) chance to jam it. Previously ECM was purchasable but did nothing.
- **Explosions** (`game.explosions`, `spawnExplosion`, `drawExplosions`) — a lightweight flash + expanding debris-shard effect on ship destruction, aged/pruned in `update()`.
- **Station no-fire zone** (`nearStationSafeZone`, `provokeStationPolice`, `spawnPoliceResponse`) — firing weapons near the station has a chance to raise `legal` and scramble a police Viper, called from `fireLaser`/`fireMissile`.
- **Escape capsule / energy bomb** — two new `EQUIPMENT` entries. Escape capsule is consumed on death for a better credit/legal outcome than the default respawn (`destroyPlayer`). Energy bomb (`useEnergyBomb`, `KeyB`) is a one-shot area attack on nearby hostiles.
- Station rotation (`drawStation`) is single-axis roll only (`rotatePoint(v, 0, 0, o.rot)`), matching the original's Coriolis-station spin — it previously combined pitch+roll into a tumble.
- **Planet/sun placement** (`buildSystemBodies`) — position, size and colour are seeded from the system itself (`hash32(sys.name + ":bodies")`), not the visit, so a system looks the same every time you return — mirroring the original's approach of deriving planet/sun placement from the system seed rather than randomising it per arrival. Planet radius scales with `sys.radius`; colour hue comes from `sys.economy` (industrial → warm/barren, agricultural → cool/lush). Distance-to-radius ratio is tuned for this engine's projection scale, not copied literally from the original's numbers (a literal port of the source's ~2.67-planet-radii station distance made the planet swallow the whole screen at launch here).
- **Planet/sun rendering** (`drawPlanetBody`, `drawSunBody`) — respects `game.graphicsMode` like everything else now: wireframe mode draws an open stroked circle + meridian/equator guides + rings with no fill (so stars/ships behind remain visible through it), solid mode draws the shaded version with no wireframe overlay. Solid-mode planets get a radial-gradient base, then all surface detail is generated as local sphere directions and projected through the camera (`surfaceEllipseDirs`, `projectedSurfacePatch`, `drawProjectedCloud`, `gasBandDirs`). This is deliberate: clouds, craters, polar caps, gas bands, storms, city clusters, rings and debris must not be painted as 2D ellipses/gradients on the projected disc, because those rotate with ship roll. The hard day/night terminator is now `drawPlanetTerminatorShadow`, which ray/sphere-samples the visible disc against the real world-space sun direction instead of drawing a screen-axis linear gradient. Night-side cities are seeded clusters of small dots (`geo.cities[].dots`) drawn only where each dot is camera-facing and on the anti-sun side. `drawPlanetAtmosphere` is a very subtle ultra-only limb halo; keep it quiet so it does not overpower the hard terminator. General object lighting is sun-ward (`collectSolidFaces`'s `lit`): low ambient, wide Lambert term, small camera-facing fill, so every ship/station has a genuinely dark side. Suns are still mostly intentional screen-facing billboards/glows; if adding anchored sun surface features, use the same projected-sphere rule rather than screen ellipses.
- **Rotating planet surface** (`drawPlanetFeatures`, plus `rotAxis`/`rotSpeed`/`rot` on the planet object) — surface blotches are real points on the sphere: seeded unit-sphere directions rotated each frame by `quatRotate(quatFromAxisAngle(axis, rot), localDir)` and projected individually via the normal `cameraTransform`/`project` pipeline, culled to the camera-facing hemisphere (`dot(worldDir, viewDir) > .08`) and foreshortened near the limb. This is deliberately the same fix pattern as the ship hull texture UV bug — object-local data rotated properly, not a screen-space billboard recomputed from a rotating normal — so the surface now actually turns with the planet instead of staying glued to the screen. Blobs use soft radial gradients (feathered edges) rather than flat-fill ellipses for a less "blotchy" look.
- **Saturn-style rings** — ~40% of planets get `hasRing` + `ringInner`/`ringOuter`/`ringFlatten`/`ringTilt` seeded in `buildSystemBodies`. `drawPlanetRing(o, r, hue, w, h, front)` draws the ring as a genuine 3D circle: points are generated in a world-space plane fixed to the planet (`o._ringBasis`, derived from `rotAxis` + `ringTilt` and cached on first use) and individually run through the normal `cameraTransform`/`project` pipeline, same as any other object — **not** `ctx.ellipse`'s rotation parameter, which is a flat 2D canvas twist that ignores the camera's own orientation and made the ring visibly rotate with the *camera* (e.g. on roll) instead of staying anchored to the planet. `front` (`true`/`false`/`null`) selects which half to draw via a real depth test (`dot(offset, viewDir)` against the planet center) rather than a guessed screen-space angle range: solid mode draws the back half first, the planet, then the front half on top so the ring correctly passes behind the body; wireframe mode passes `null` and draws the whole loop in one pass since nothing is opaque there anyway. **Sun-shaded rings with a real dark side** (`ringShade`, 1 → ~0.1 brightness): the hard near-black core is the planet's true umbra cylinder (anti-sun side, perpendicular distance from the sun axis `< planetR`), but the umbra *alone* is misleading on screen — most of the eclipsed arc hides behind the planet disc, leaving just two thin dark stubs poking out either side, which reads as a bug ("two stripes"), not a shadow. So `ringShade` also applies a smooth falloff across the entire night half of the ring (`-along/d` from 0.95 at the terminator plane to 0.3 fully anti-sun), giving the ring a whole dark side consistent with the planet terminator. Since a stroked path can't change colour mid-stroke, the shade is quantised to 8 levels and each band flushes/restarts its path when the level changes (as well as at gaps and the front/back split), with saturation and lightness both scaled down as shade drops; debris-ring particles multiply their colour by the same `ringShade` value. Shading is solid/both-mode only — wire mode keeps flat 1984 ring lines.
- **Ship yaw/pitch from velocity** (`orientShipToVelocity`) — `rotatePoint` applies yaw (around Y) then pitch (around X) to the model's local +Z nose, giving `forward' = (-sin(yaw), -cos(yaw)*sin(pitch), cos(yaw)*cos(pitch))`. Solving that for `forward' == normalize(v)` requires `yaw = atan2(-v.x, hypot(v.y, v.z))` and `pitch = atan2(-v.y, v.z)` — note yaw groups `vx` with `hypot(vy,vz)`, not `hypot(vx,vz)`, and pitch uses plain `vz`, not a hypot. A previous fix attempt (`atan2(-v.x,-v.z)` / `atan2(-v.y,hypot(v.x,v.z))`) matched `normalize(v)` for axis-aligned test vectors (pure X, pure Y) but silently flipped the nose for general 3D velocity — the axis-aligned cases pass by coincidence because they zero out the term that's actually wrong. Always verify an Euler-angle fix against a vector with all three components nonzero.
- **Weapon fire is now visible and aimed at the actual target** (`game.enemyShots`, `game.playerShot`, `fireAtPlayer`, `fireAtShip`, `drawEnemyShots`) — previously a hostile's attack was a shields hit and a text message with no visual, and the player's own beam was hardcoded to draw from bottom-center to the exact screen center regardless of what was actually hit. Both are now real: `laserPower()` carries a `color` per weapon type (Pulse/Beam/Military — BBC Master changes laser sight colour per type, confirmed on elite.bbcelite.com) and `fireLaser` stores the hit target's position in `game.playerShot` so the beam terminates there, falling back to the crosshair only on a miss or when the target is the player (who has no visible ship in this first-person view to aim at). `game.enemyShots`/`fireAtShip` use the same `to: null → crosshair, else → target position` convention. Aged/pruned in `update()` alongside `game.explosions`.
- **Hostiles can target other ships, not just the player** (`o.target`, `assignHostileTarget`, `pickPreyFor`, `pickHostileFor`, `hostileTargetPos`, `fireAtShip`) — `o.target` is `null` (meaning the player) or a live reference to another ship object. Police prefer the nearest non-police hostile within 1800 units (falling back to the player); pirates/aliens have a 55% chance to prefer the nearest non-hostile ship within 1400 units over the player (aliens/`thargoid` always target the player, kept deliberately relentless). Re-evaluated when the current target dies, leaves `game.objects`, or drifts past 2200 units away. All the distance-banded steering/firing math in `updateShipAI` reads from `hostileTargetPos(o)` instead of a hardcoded `game.camera.pos`. A hunted non-hostile ship (`o.huntedBy`, set when it's picked as prey) flees that specific attacker rather than only reacting to player proximity — flee direction/despawn distance are measured from `o.fleeFrom`, a position snapshot taken when flee starts (not a live reference, so it survives the threat being destroyed mid-flight). `destroyObject(o, byPlayer)` only grants the player kills/credits/legal-status changes when `byPlayer` is true — `fireAtShip` passes `false` so NPC-vs-NPC kills don't reward the player.
- Stars (`drawStars`) now carry a per-star seeded hue/size and a sine-modulated twinkle instead of being uniform flat-white 1px squares.
- **Ship identification** — hostility is no longer signalled by recoloring the ship's wireframe/hull red (`shipColor` now only varies by faction: police cyan, aliens violet, everyone else neutral off-white). Instead `drawShipLabels` draws a small HUD box above each nearby ship with its name (reusing `o.name`, already "PIRATE MAMBA" etc.) and a colour-coded status line (`shipStatus`: HOSTILE/PURSUING/FLEEING/PATROL/CRUISING, derived from `o.hostile`/`o.police`/`o.aiMode`). The 3D scanner's red-for-hostile blips were left alone — that's a distinct radar-style UI, not the "highlighting" this replaced.
- **Combat AI is distance-banded** (`updateShipAI`'s hostile branch) — echoing the source's tactics routine ("ships that are too close turn away" / appropriately-ranged ships close in): under ~260 units a hostile peels off and opens distance, 260-900 it orbits/strafes perpendicular to the line-of-sight while roughly holding range, beyond 900 it closes in on a lead point ahead of the player. Before this fix, hostiles always steered toward a point near the player's nose regardless of current distance, so they flew straight through the player on a collision course instead of dogfighting — and sat at point-blank range where the laser hit-test (`rel.z < 30`) excludes them entirely, which is why lasers seemed to do nothing. Verified via the live game loop (not a synthetic dt-stepped loop — `performance.now()`-driven jink terms need real wall-clock time to oscillate correctly, a synthetic tight loop with a fixed dt will falsely look like linear drift/instability).
- **Per-class combat speed** — `SHIP_STATS` entries now carry a `speed` multiplier (relative to Cobra Mk III = 1.0), derived from the original BBC max-speed table (`Max. speed / 28`) instead of every hostile sharing one flat speed cap. Applied in both the attack and flee branches of `updateShipAI`.
- **Nose-led ship movement** (`steerVelocity`, `shipForward`, `yawPitchForDirection`) — NPC ships now mimic the original `MVEIT`/`TACTICS` shape: AI still produces a desired vector, but `steerVelocity` turns the ship's nose toward it, manages a scalar `o.speed`, then moves along `shipForward(o)`. The previous model changed `o.vel` directly and rotated the mesh afterward, which made ships slide like drones rather than fly forward through their turns. Near-zero desired vectors fall back to the current nose direction, preserving the old anti-jitter guard for formation ships.
- **Ship roll is local to the nose** (`rotateShipPoint`, `setNoseDirection`) — moving objects use a ship-specific transform where roll banks around the yaw/pitch nose vector instead of screen/world Z. This keeps the rendered nose aligned with `shipForward(o)`; otherwise a banked ship or missile can appear to travel sideways even when the velocity math is correct. New ships and armada entries seed their attitude from initial velocity, and missiles now steer nose-first rather than snapping velocity directly toward the target.
- **Player steering has rate inertia** (`rollRate`, `pitchRate`, `yawRate`, `updatePlayerControls`) — keyboard controls now follow the original `JSTX`/`JSTY` idea from `BUMP2`/`REDU2`: key presses accelerate a turn rate and releasing keys damps it back toward centre. Roll ramps faster and higher than pitch, matching the original's finer pitch control; yaw uses the same browser-era rate model even though BBC Elite did not have yaw.
- **Wire mode is hidden-line, not literal wireframe** (`faceByNormalIndex`, `collectWireFaceVisibility`, `wireEdgeVisible`, `drawWireOccluderFaces`) — mirrors the original `LL9` strategy: each edge knows its two adjacent faces, face visibility is computed first, then an edge is drawn only if at least one attached face is visible. Visible hull faces are also filled black before stroking the wire edges, so nearer wireframe objects mask farther objects/stars while staying visually "wire". Applies to ships and stations; models without face metadata still fall back to drawing all edges.
- **Armada spawn distance**: the formation's actual spawn position is `local.z` in `spawnArmada` (currently `3800 + ring*340 + (slot%2)*80`, i.e. ~5x the original 760 base) — note this is a *different* variable from `center` (also used, for the orbit-target/initial-velocity-direction math), so changing one without the other will desync spawn position from orbit behavior. The general ship-culling distance in `update()` was bumped from 3800 to 6500 to accommodate — it was previously despawning armada ships within the same frame they spawned, since 3800 was both the cull radius and roughly the new spawn distance.
- **Ship HUD hp/shield bar** — `drawShipLabels` now draws a small colour-graded bar (green/amber/red by remaining hp fraction) under each ship's name/status text, in addition to the existing text.
- **Local & galaxy star charts** (`renderGalaxyMap`, `renderLocalMap`, `game.mapMode`) — the Chart panel is inline SVG rather than a text table, like the original's two chart screens. Galaxy map plots every system in the current galaxy at its seeded `(x,y)`; local map is centered on the current system with distance rings (a dashed one at the 7 LY jump range) and only labels systems within ~8 LY by name — Lave alone has 100+ neighbours inside 24 LY, so labelling everything in range is unreadable. Clicking a `.map-dot` sets `game.target`, replacing the old per-row Select buttons.
- **Biome-driven planet surface** (`planetBiomeForSystem`, `biome` on the planet object from `buildSystemBodies`) — economy/description now pick ocean, desert, rocky, industrial, or gas-giant treatments. Ocean worlds get seeded continents/islands and projected spiral cloud systems; industrial worlds get dark polluted land, projected industry patches, debris rings/fields, and dense amber night-city dot clusters; rocky/desert worlds get craters/ridges; gas giants get projected latitude bands and a storm patch. The day/night shadow is `drawPlanetTerminatorShadow`, not a screen-space gradient.
- **Ship scale now follows the original source stats** (`SHIP_STATS`, `shipStats`) — cross-checking our `MODELS` vertex data against the source confirmed the meshes themselves use the original coordinate units, while the previous collision/scale table was hand-guessed and compressed the size range. `SHIP_STATS.r` is now the source targetable-area radius, `hp` is max energy, and `speed` is source max speed normalized to Cobra Mk III = 1.0; `shipStats()` only falls back to `modelBoundingRadius()` for models without source stats.
- **Combat-orbit direction no longer flips** (`updateShipAI`'s mid-range band, `o.orbitSign`) — the orbiting-at-range steering used to scale the entire tangential direction by `jink` (a `Math.sin(...)` term peaking at +-130, i.e. *larger* than the 30-unit inward-pull term it was added to), so every time the sine crossed zero (~every 2.8s) the ship's whole intended heading reversed instead of continuing to circle the same way — visible as ships flipping direction and barely moving before flipping again, rather than streaking around in a smooth orbit. Fixed by giving each ship a fixed `orbitSign` (chosen once, lazily) that sets a constant rotational direction, with `jink` now only breathing the orbit radius in and out. Verified against the live game loop: velocity magnitude now holds steady (~65-68) throughout instead of collapsing toward zero at each flip, and the velocity vector sweeps smoothly through a full rotation instead of reversing.
- **Object and station geometry comes from the OBJ/source pass** (`MODELS` before `SHIP_STATS`) — missile, escape pod, alloy plate, asteroid, boulder, splinter, rock hermit, Coriolis, and Dodo now use the OBJ folder's vertex/line data with BBC source edge-face/normal metadata where needed so solid rendering can cull and shade faces. The OBJ files contain `v`/`l` data only, so source face metadata is still required for shaded surfaces; the alloy plate remains a special two-sided quad because the original source treats it as a one-face object.
- **Missiles are real tracked projectiles, not hit-scan** (`spawnMissile`, `updateMissile`, `detonateMissile`) — both player (`fireMissile`) and NPC (`fireAtPlayer`'s missile branch) firing spawn an actual `game.objects` entry with `type: "missile"` that homes via simple pursuit, orients its nose with the existing `orientShipToVelocity`, and detonates on proximity or fuse expiry. ECM still jams NPC missiles at the same 75% rate as before, just applied at spawn/detonation instead of at instant-hit time.
- **Escape pods, alloy plates, and destructible debris** (`destroyObject`, `spawnEscapePod`, `scoopCargo`) — on a player kill there's a 20% chance the pilot ejects (`spawnEscapePod`) instead of the ship simply exploding: this is strictly better for the target (they survive) and worse for the player (no bounty), so it's a real tradeoff rather than a reskin. Non-eject kills have a 50% chance to drop debris, now split between the existing cargo canister and a new alloy plate (`type: "plate"`) — both are scoopable via `scoopCargo`, which now also handles the plate's fixed "Alloys" cargo type vs. canisters' random good.
- **Asteroid/boulder/splinter mining loop + Mining Laser** (`spawnAsteroid`, `spawnSplinter`, `spawnMineralCanister`, `EQUIPMENT.miningLaser`) — asteroids are the mineable rock type: shooting one apart (`destroyObject`'s `"asteroid"` branch) spawns 2-3 splinters plus a chance of an immediate mineral canister, and splinters themselves have a further chance to drop a mineral canister when destroyed. Boulders are deliberately inert — same rock-tumbling physics and laser-targetable, but no reward on breakup — a hazard/filler, not a resource. Mineral canisters use a `forcedGood` field (Minerals/Gold/Platinum/Gem-Stones) so `scoopCargo` gives a specific mining payout instead of a random good. The Mining Laser equipment doubles laser damage specifically against rock-type targets (`fireLaser`'s `isRockTarget` check), making dedicated mining meaningfully faster without buffing anti-ship combat.
- **Rock Hermit as a second dockable location** (`dockableTarget`, `type: "hermit"`) — `dockableTarget()` generalizes docking to the nearest `station`-or-`hermit` object, used by both `dock()` and `updateAutoDock()`; `game.dockedAt` (set on successful dock) drives the panel subtitle and the docked-view backdrop text instead of both being hardcoded to `${system} STATION`. Unlike the station, a hermit doesn't require slot-rotation alignment to dock (`dock()`'s `slotOk` check is skipped for non-`station` types) since it's a static rock outpost, not a spinning bay. Presence is deterministic per system (`mulberry(hash32(sys.name + ":hermit")) < .3`, ~30% of systems), matching the existing pattern of seeding system-level dressing from the system itself rather than per-visit RNG.

- **`game.stars` doubles as motion-sensation dust, so it now actually streaks** (`drawStars`) — this same star field is what `update()` recycles ahead of the camera as it moves (see the `rel.z`/`rel.x`/`rel.y` bounds check in the main flight-control block), so it's the game's only "space dust," not just decorative background; the initial brightness/count/glow pass above still rendered every star as a plain dot regardless of speed, so it read as twinkling stars rather than dust rushing past. Fixed by stroking each star as a short line from `(pr - dir*streak)` to `pr` instead of a `fillRect`, where `dir` is the unit vector from screen centre to the star's projected point (the classic radiate-from-the-vanishing-point technique, same idea as `drawTransitionEffect`'s warp streaks) and `streak = speedFrac * closeness * 30` (`speedFrac = game.speed / game.maxSpeed`, `closeness` from the star's own depth, matching the existing brightness falloff). At `speed = 0` the line's two endpoints coincide, and since `ctx.lineCap = "round"` is already set by `drawSpace` before `drawStars` runs, a zero-length round-capped stroke renders as the same plain dot as before — so idle/docked view is unaffected and the streak only appears once actually moving, growing with speed.
- **Dust streaks fade at the tail instead of ending in a hard edge** (`drawStars`) — the streak line's `strokeStyle` is now a `ctx.createLinearGradient` from the tail (fully transparent) to the head/current position (the star's normal brightness/twinkle alpha), built from an rgba/hsla colour string per star rather than relying on `ctx.globalAlpha` (which can't vary along a single stroke). `shadowColor` for the existing glow still uses the solid head-alpha colour, since canvas shadows don't support gradients.
- **Fixed recycled dust landing outside the view after turning, thinning the field over time** (main flight-control block in `update()`) — when a star left the camera-relative frustum it was recycled to `camera.pos + world-space offset` (jitter mostly along world +Z), a fixed world-axis box that only lines up with "ahead of the ship" if the camera happens to still be facing the original identity orientation; once the player rotates away from that (normal play, not an edge case), newly recycled stars increasingly land outside the actual current view direction instead of in front of the ship, so the field can visibly thin out over a flight rather than replenishing evenly. Fixed by building the recycle offset with `cameraLocalToWorld(game.camera, vec(...))` (already used elsewhere for camera-relative placement) so new dust is generated in the camera's *current* local forward box, then rotated into world space, regardless of how much the ship has turned. Verified by simulating hundreds of recycle steps under sustained turning + max speed and confirming the in-frustum star count holds steady instead of decaying.
- **Dashboard panel padding tightened to stop clipping** (`.dashboard`, `.dash-section`, `.bar-row`, `.bar`, `.row`/`.stat-line`, new `.dash-section .row`) — the dashboard strip is a fixed 104px CSS Grid row (`#app`'s `grid-template-rows`), and `.dashboard`+`.dash-section` padding (8px each, stacked) plus generous per-row spacing were eating most of that budget: the 3D Scanner's hard-coded 86px-tall `.scanner-wrap` didn't fit the ~72px actually available and was being silently cut off by `.dash-section`'s `overflow: hidden`, and the Weapons section's 4 `.row` entries (Laser/Missiles/ECM/Cargo, ~26px each) totalled 104px against an ~86px content budget, clipping the bottom "Cargo" row. Fixed by halving `.dashboard`/`.dash-section` padding to 4px, trimming `.bar-row` margin and `.bar` height slightly, and adding a scoped `.dash-section .row` override (smaller padding + font-size) so the Weapons rows shrink without affecting `.row`'s shared use elsewhere (e.g. the Status panel, which has room to spare outside the fixed-height dashboard strip). Verified by measuring actual DOM `getBoundingClientRect()`s live rather than eyeballing a screenshot — the fixed-height strip makes "looks fine at this window size" an unreliable check.

- **View switching: fwd/rear/left/right cockpit views + external chase camera** (`game.viewMode`, `VIEW_YAW`/`VIEW_LABELS`, `viewCamera`, `setViewMode`; arrows select cockpit views, `V` toggles external) — the simulation (steering, weapons, scanner, dust recycling) always runs off the real `game.camera`; `viewCamera()` derives the rendered camera (local-yaw composed quats for rear/left/right, a hull-radius-scaled chase offset pitched down ~10° for external) and `drawSpace` swaps it in for the duration of the frame only. The player's ship is rendered only in external view, as a pseudo-object carrying `quat` (the flight camera's own orientation) — `drawWireModel`/`collectSolidDetails` accept `o.quat` as an alternative to Euler `rot/pitch/roll` for exactly this. Lasers are forward-mounted (firing blocked outside fwd view, like the original's per-view laser fittings), the CSS crosshair and the player-shot beam overlay are fwd-only, dust streaks match each view's apparent motion (radial fwd/ext, inward rear, horizontal side sweep), and `setViewMode("fwd")` is called at dock-complete/respawn/commander-load so a flight never starts looking backwards. `viewMode` is deliberately not in `SAVE_FIELDS`.

- **Procedural big-cat-stripe ship decals** (`buildDecalTexture`, `getShipDecal`, `decalCache`, `DECAL_STYLES`, `drawUVLayer`) — every ship gets a seeded tiger-stripe livery multiplied over the brushed-metal pass via the same fixed face-UV mapping (so it stays glued to the hull). Style comes from the local system (government → paint discipline/jitter, economy → wealth: rich systems get sparse crisp stripes + a gold accent) and role (police cyan, pirate heavy black, alien violet, trader/private muted). Battle-damage scorch blotches appear by hp band (0/1/2 at <75%/<45% hp), keyed into the cache so the texture regenerates as a ship takes damage; `decalSeed` is set at spawn (falls back to `hash32(name+model)` for the player/armada). LRU-capped cache (64).
- **Eject effects** (`spawnEjectFlash`, the `"eject"` explosion kind, `drawEjectSequence` + the `"eject"` transition kind) — NPC pilot ejections pop a cold cyan ring + glass glints alongside a reduced explosion; the player's own escape-capsule death now plays a 1.6s transition beat (white canopy-blow flash, tumbling rotating streaks, closing vignette) before `destroyPlayer`'s respawn state change runs in `onComplete`, same pattern as launch/dock/jump. Uncushioned (no capsule) deaths stay abrupt on purpose.
- **Old School / Ultra Elite presentation presets** (`game.fxLevel`, `applyPresetMode`, buttons in the Ship panel next to the graphics-mode row, persisted in `SAVE_FIELDS`) — "classic" = wireframe + flat uniform star dots (no hue/twinkle/glow/streaks) + no metal/decal/glass, labels off: the 1984 look; "ultra" = both-mode + everything on. Implemented as gates in `drawStars`/`drawFaceTexture`/`glassGradient`/`getShipDecal`, so the existing individual toggles (G key, graphics buttons, L) still work on top of either preset.
- **NPC comms chatter + HUD comms log** (`commsMessage`/`renderComms`/`hail`, `#commsLog` overlay bottom-left of the flight view, line-generator functions `hostileThreatLine`/`fleeLine`/`policeScrambleLine`/`witnessLine`/`ambientChatterLine`) — situation hooks: first hostile attack (once per ship via `o._hailed`), all three flee triggers, police scramble, witnessed player kills, docking clearance + auto-dock acknowledgment from station control, plus an ambient timer (`game.commsClock`, every 16-42s in flight) that picks a live ship and generates role/system-aware gossip (real goods, destination, economy/government names). Lines expire after ~9.5s, max 4 shown, HTML-escaped. This panel is the intended home for player chat if multiplayer ever lands.

## Remaining gaps vs. the original
