# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A browser-based Elite clone ("Vector Elite"), currently implemented entirely in a single file, `index.html` (~2,600 lines: CSS, then one `<script>` block). No build step, no package manager, no dependencies â€” it's an HTML file you open or serve directly. There is no test suite and no linter configured.

This is a for-fun vibe-coding experiment: the goal is a modern-spin take on 1984 Elite, built almost entirely through conversational iteration rather than a planned spec. Priorities are "does it feel good to fly/fight/trade" and "is it fun to keep bolting features on," not production hygiene. Keeping everything in one `index.html` has been a nice constraint so far (paste-and-run, no tooling), but it's not sacred â€” separate asset files (sprites, textures, sound, even a split-out JS module or two) are fine if a feature genuinely wants them. Don't over-engineer or add process (build steps, test harnesses, frameworks) unless it's clearly earning its keep.

## Commands

- **Run it**: open `index.html` directly in a browser, or serve it locally (`npx serve .` or `python3 -m http.server`) â€” a static server avoids any `file://` restrictions.
- **No build/lint/test commands exist.** Verify changes by loading the page in a browser and checking the console; there's nothing to compile.

## Architecture

Everything lives in `index.html`, in this rough top-to-bottom order:

1. **CSS** â€” CRT/vector-scope aesthetic (`:root` custom properties for the green/amber/cyan palette). Layout is CSS Grid: `#app` splits into the flight view + side panel (`.main`), the dashboard strip, and the message/command footer. Two `@media` breakpoints (860px, 520px) collapse the layout for mobile.

2. **Math/camera layer** (`vec`, `quat*` functions, `rotatePoint`, `makeCamera`, `cameraTransform`, `forwardVector`) â€” a quaternion-based camera. The player's ship never moves; instead `game.camera` (position + orientation quaternion) moves through a static `game.objects` list, and world positions are transformed into camera space for projection. `project()` does the perspective divide into screen space.

3. **Ship blueprints** (`MODELS`, `buildBlueprint`) â€” each ship is defined as `verts`/`edges`/`edgeFaces`/`normals` (+ optional `details` for windows/engines/trim). `buildBlueprint()` derives polygon `faces` from the edge/face-adjacency data at load time, ordering vertices by angle around each face's centroid and correcting winding via `faceNormal`. If you add a new ship, you only need to supply verts/edges/edgeFaces/normals â€” faces are computed, not authored by hand. Any ship can also be overridden at runtime by a hand-edited `.obj` file in `ship_models/` â€” see the note under "Notable systems" below. This is the one deliberate exception to "everything lives in index.html": real geometry editing needs a real 3D tool, not more JS literals.

4. **Galaxy generation** (`GalaxyFactory`, `galaxies`) â€” deterministic, seeded system generation modeled on the original Elite's algorithm: three 16-bit seed words are "twisted" per system to derive economy/government/tech/population/name/species. All 8 galaxies for the session are generated up front at module load (`Array.from({ length: 8 }, ...)`). `LAVE_INDEX`/`DISO_INDEX` are found by name rather than hardcoded indices.

5. **Game state** â€” a single mutable `game` object holds all mutable state (position, credits, cargo, equipment, current/target system, camera, `objects` array of in-flight entities, etc). There's no state management abstraction; functions read/write `game.*` directly.

6. **Simulation** (`update`, `updateShipAI`, `steerVelocity`, `updateStation`, `updateAutoDock`) â€” called once per animation frame with `dt`. NPC ships pick an `aiMode` (`cruise` / `orbit` / `formation` / `attack`) and `steerVelocity` moves them toward a per-mode target position. Combat, docking legality, fuel/shield/energy regen, and legal-status decay all happen here.

7. **Rendering** (`drawSpace`, `drawWireModel`, `drawStation`, `collectSolidFaces`/`collectSolidDetails`, `drawSolidItems`) â€” two graphics modes selected by `game.graphicsMode`: `"wire"` draws model edges directly; `"solid"` back-face-culls and flat-shades polygons against a light direction derived from the sun's position (`starLightDirection`), then z-sorts and fills them. The scanner (`drawScanner`) is a separate small canvas driven by the same `game.objects`.

8. **UI panels** (`renderPanel`, `renderMarket`/`renderChart`/`renderEquip`/`renderStatus`/`renderShip`/`renderHelp`/`renderCheats`, `attachPanelEvents`) â€” the side panel is plain string-templated HTML re-rendered into `panelBody.innerHTML` on every state change; `attachPanelEvents()` re-binds click handlers after each render since the DOM is fully replaced. There's no virtual DOM / diffing â€” this is intentional and fine at this scale, just be aware a full panel re-render happens on most actions.

9. **Input** â€” a single `keydown`/`keyup` listener maintains a `keys` Set for held movement keys (WASD/QE/+-), plus one-shot key actions (space to fire, J/O/C/M/R/G/1-4) handled inline in the `keydown` handler. Footer buttons call the same functions as their keyboard shortcuts.

10. **Persistence** â€” `saveGame`/`loadGame` serialize a subset of `game` to `localStorage` under `vectorEliteSave`. `loadGame` has one hardcoded migration check for saves predating `galaxySeedVersion` (remaps `current`/`target` from raw indices 0/1 to the found Lave/Diso indices).

## Notable systems (ported from / inspired by the BBC Master source)

- **Hull texturing** (`buildHullTexture`, `getHullTexture`, `drawTexturedTriangle`, `drawFaceTexture`) â€” solid-shaded faces get a real texture-mapped brushed-metal pass, not just a flat lit fill. UV coordinates are computed **once, from static object-local vertex positions** â€” `buildBlueprint` computes `model.faceUV` alongside `model.faces` (reusing the same per-face `u`/`v` basis already used for winding order); the station uses the exact same `buildBlueprint` pipeline (see below) so it gets this for free too. This matters: UV must **not** be derived from the rotating camera-space normal per frame â€” that basis isn't rigidly attached to the object (picture combing a hairy ball) and makes the texture swim/flip independently of the actual rotation instead of staying glued to the hull. `drawTexturedTriangle` is the classic canvas affine-transform trick: fan-triangulate the face, solve a 2x2 linear system per triangle to map the fixed local UV space onto current screen space, clip, `drawImage`. Composited with `multiply` blending over the existing flat-shaded fill so per-face lighting is preserved. Culled below a screen-space area threshold for performance.
- **Station model** (`STATION_MODEL`, `STATION_SIZE`) â€” the Coriolis station's octagonal-prism hull is a `buildBlueprint()` model like every ship, not bespoke inline verts/edges/faces. Since its 10 faces (2 octagon caps + 8 side quads) are simple, `edgeFaces` is hand-authored directly (mapping which of the 10 faces each of the 24 edges borders) rather than relying on any derivation â€” `buildBlueprint` still does the winding/UV work from that adjacency data. Verts are baked to real hull scale (`* STATION_SIZE`) up front so the shared UV/texture pipeline needs no station-specific scale handling. `drawStation` stays a separate function from `drawWireModel` (not just reuse) because the station's rotation convention is single-axis roll (`o.rot` passed as `roll`), not the yaw/pitch/roll a ship uses, and it has the extra docking-slot-marker overlay ships don't.
- **AI roles & flee behavior** (`updateShipAI`, `fireAtPlayer`) â€” ships carry an explicit `role` (`"trader"|"pirate"|"police"`) and `maxHp` set at spawn. Traders that get spooked flee rather than fight; hostile non-police ships below ~32% hp break off and flee off-screen. This replaces the old behavior where a startled trader would flip straight into an attacker.
- **ECM and missiles** â€” hostile ships now sometimes fire a missile instead of a laser ping; `game.equipment.ecm` gives a real (75%) chance to jam it. Previously ECM was purchasable but did nothing.
- **Explosions** (`game.explosions`, `spawnExplosion`, `drawExplosions`) â€” a lightweight flash + expanding debris-shard effect on ship destruction, aged/pruned in `update()`.
- **Station no-fire zone** (`nearStationSafeZone`, `provokeStationPolice`, `spawnPoliceResponse`) â€” firing weapons near the station has a chance to raise `legal` and scramble a police Viper, called from `fireLaser`/`fireMissile`.
- **Escape capsule / energy bomb** â€” two new `EQUIPMENT` entries. Escape capsule is consumed on death for a better credit/legal outcome than the default respawn (`destroyPlayer`). Energy bomb (`useEnergyBomb`, `KeyB`) is a one-shot area attack on nearby hostiles.
- Station rotation (`drawStation`) is single-axis roll only (`rotatePoint(v, 0, 0, o.rot)`), matching the original's Coriolis-station spin â€” it previously combined pitch+roll into a tumble.
- **Planet/sun placement** (`buildSystemBodies`) â€” position, size and colour are seeded from the system itself (`hash32(sys.name + ":bodies")`), not the visit, so a system looks the same every time you return â€” mirroring the original's approach of deriving planet/sun placement from the system seed rather than randomising it per arrival. Planet radius scales with `sys.radius`; colour hue comes from `sys.economy` (industrial â†’ warm/barren, agricultural â†’ cool/lush). Distance-to-radius ratio is tuned for this engine's projection scale, not copied literally from the original's numbers (a literal port of the source's ~2.67-planet-radii station distance made the planet swallow the whole screen at launch here).
- **Planet/sun rendering** (`drawPlanetBody`, `drawSunBody`) â€” respects `game.graphicsMode` like everything else now: wireframe mode draws an open stroked circle + meridian/equator ellipses + rings with no fill (so stars/ships behind remain visible through it), solid mode draws the shaded version with no wireframe overlay (the meridian ellipses are wire-mode only â€” drawing them on top of the solid fill looked like a wireframe ghost showing through the sphere, so solid mode relies purely on the gradient + rotating features for shape, same as how ships don't show edge wireframes over their solid-shaded faces). Solid-mode planets get a radial-gradient sphere with a terminator shaded toward the system's sun, plus rotating surface features (see below). Suns get a warm radial-gradient glow with a shimmering edge, inspired by (not a literal port of) the original's randomised-half-width horizontal-line shimmer technique; wire-mode suns are just a stroked circle + faint corona.
- **Rotating planet surface** (`drawPlanetFeatures`, plus `rotAxis`/`rotSpeed`/`rot` on the planet object) â€” surface blotches are real points on the sphere: seeded unit-sphere directions rotated each frame by `quatRotate(quatFromAxisAngle(axis, rot), localDir)` and projected individually via the normal `cameraTransform`/`project` pipeline, culled to the camera-facing hemisphere (`dot(worldDir, viewDir) > .08`) and foreshortened near the limb. This is deliberately the same fix pattern as the ship hull texture UV bug â€” object-local data rotated properly, not a screen-space billboard recomputed from a rotating normal â€” so the surface now actually turns with the planet instead of staying glued to the screen. Blobs use soft radial gradients (feathered edges) rather than flat-fill ellipses for a less "blotchy" look.
- **Saturn-style rings** â€” ~40% of planets get `hasRing` + `ringInner`/`ringOuter`/`ringFlatten`/`ringTilt` seeded in `buildSystemBodies`. `drawPlanetRing(o, r, hue, w, h, front)` draws the ring as a genuine 3D circle: points are generated in a world-space plane fixed to the planet (`o._ringBasis`, derived from `rotAxis` + `ringTilt` and cached on first use) and individually run through the normal `cameraTransform`/`project` pipeline, same as any other object â€” **not** `ctx.ellipse`'s rotation parameter, which is a flat 2D canvas twist that ignores the camera's own orientation and made the ring visibly rotate with the *camera* (e.g. on roll) instead of staying anchored to the planet. `front` (`true`/`false`/`null`) selects which half to draw via a real depth test (`dot(offset, viewDir)` against the planet center) rather than a guessed screen-space angle range: solid mode draws the back half first, the planet, then the front half on top so the ring correctly passes behind the body; wireframe mode passes `null` and draws the whole loop in one pass since nothing is opaque there anyway.
- **Ship yaw/pitch from velocity** (`orientShipToVelocity`) â€” `rotatePoint` applies yaw (around Y) then pitch (around X) to the model's local +Z nose, giving `forward' = (-sin(yaw), -cos(yaw)*sin(pitch), cos(yaw)*cos(pitch))`. Solving that for `forward' == normalize(v)` requires `yaw = atan2(-v.x, hypot(v.y, v.z))` and `pitch = atan2(-v.y, v.z)` â€” note yaw groups `vx` with `hypot(vy,vz)`, not `hypot(vx,vz)`, and pitch uses plain `vz`, not a hypot. A previous fix attempt (`atan2(-v.x,-v.z)` / `atan2(-v.y,hypot(v.x,v.z))`) matched `normalize(v)` for axis-aligned test vectors (pure X, pure Y) but silently flipped the nose for general 3D velocity â€” the axis-aligned cases pass by coincidence because they zero out the term that's actually wrong. Always verify an Euler-angle fix against a vector with all three components nonzero.
- **Weapon fire is now visible and aimed at the actual target** (`game.enemyShots`, `game.playerShot`, `fireAtPlayer`, `fireAtShip`, `drawEnemyShots`) â€” previously a hostile's attack was a shields hit and a text message with no visual, and the player's own beam was hardcoded to draw from bottom-center to the exact screen center regardless of what was actually hit. Both are now real: `laserPower()` carries a `color` per weapon type (Pulse/Beam/Military â€” BBC Master changes laser sight colour per type, confirmed on elite.bbcelite.com) and `fireLaser` stores the hit target's position in `game.playerShot` so the beam terminates there, falling back to the crosshair only on a miss or when the target is the player (who has no visible ship in this first-person view to aim at). `game.enemyShots`/`fireAtShip` use the same `to: null â†’ crosshair, else â†’ target position` convention. Aged/pruned in `update()` alongside `game.explosions`.
- **Hostiles can target other ships, not just the player** (`o.target`, `assignHostileTarget`, `pickPreyFor`, `pickHostileFor`, `hostileTargetPos`, `fireAtShip`) â€” `o.target` is `null` (meaning the player) or a live reference to another ship object. Police prefer the nearest non-police hostile within 1800 units (falling back to the player); pirates/aliens have a 55% chance to prefer the nearest non-hostile ship within 1400 units over the player (aliens/`thargoid` always target the player, kept deliberately relentless). Re-evaluated when the current target dies, leaves `game.objects`, or drifts past 2200 units away. All the distance-banded steering/firing math in `updateShipAI` reads from `hostileTargetPos(o)` instead of a hardcoded `game.camera.pos`. A hunted non-hostile ship (`o.huntedBy`, set when it's picked as prey) flees that specific attacker rather than only reacting to player proximity â€” flee direction/despawn distance are measured from `o.fleeFrom`, a position snapshot taken when flee starts (not a live reference, so it survives the threat being destroyed mid-flight). `destroyObject(o, byPlayer)` only grants the player kills/credits/legal-status changes when `byPlayer` is true â€” `fireAtShip` passes `false` so NPC-vs-NPC kills don't reward the player.
- Stars (`drawStars`) now carry a per-star seeded hue/size and a sine-modulated twinkle instead of being uniform flat-white 1px squares.
- **Ship identification** â€” hostility is no longer signalled by recoloring the ship's wireframe/hull red (`shipColor` now only varies by faction: police cyan, aliens violet, everyone else neutral off-white). Instead `drawShipLabels` draws a small HUD box above each nearby ship with its name (reusing `o.name`, already "PIRATE MAMBA" etc.) and a colour-coded status line (`shipStatus`: HOSTILE/PURSUING/FLEEING/PATROL/CRUISING, derived from `o.hostile`/`o.police`/`o.aiMode`). The 3D scanner's red-for-hostile blips were left alone â€” that's a distinct radar-style UI, not the "highlighting" this replaced.
- **Combat AI is distance-banded** (`updateShipAI`'s hostile branch) â€” echoing the source's tactics routine ("ships that are too close turn away" / appropriately-ranged ships close in): under ~260 units a hostile peels off and opens distance, 260-900 it orbits/strafes perpendicular to the line-of-sight while roughly holding range, beyond 900 it closes in on a lead point ahead of the player. Before this fix, hostiles always steered toward a point near the player's nose regardless of current distance, so they flew straight through the player on a collision course instead of dogfighting â€” and sat at point-blank range where the laser hit-test (`rel.z < 30`) excludes them entirely, which is why lasers seemed to do nothing. Verified via the live game loop (not a synthetic dt-stepped loop â€” `performance.now()`-driven jink terms need real wall-clock time to oscillate correctly, a synthetic tight loop with a fixed dt will falsely look like linear drift/instability).
- **Per-class combat speed** â€” `SHIP_STATS` entries now carry a `speed` multiplier (relative to Cobra Mk III = 1.0), derived from the original BBC max-speed table (`Max. speed / 28`) instead of every hostile sharing one flat speed cap. Applied in both the attack and flee branches of `updateShipAI`.
- **Nose-led ship movement** (`steerVelocity`, `shipForward`, `yawPitchForDirection`) â€” NPC ships now mimic the original `MVEIT`/`TACTICS` shape: AI still produces a desired vector, but `steerVelocity` turns the ship's nose toward it, manages a scalar `o.speed`, then moves along `shipForward(o)`. The previous model changed `o.vel` directly and rotated the mesh afterward, which made ships slide like drones rather than fly forward through their turns. Near-zero desired vectors fall back to the current nose direction, preserving the old anti-jitter guard for formation ships.
- **Ship roll is local to the nose** (`rotateShipPoint`, `setNoseDirection`) â€” moving objects use a ship-specific transform where roll banks around the yaw/pitch nose vector instead of screen/world Z. This keeps the rendered nose aligned with `shipForward(o)`; otherwise a banked ship or missile can appear to travel sideways even when the velocity math is correct. New ships and armada entries seed their attitude from initial velocity, and missiles now steer nose-first rather than snapping velocity directly toward the target.
- **Player steering has rate inertia** (`rollRate`, `pitchRate`, `yawRate`, `updatePlayerControls`) â€” keyboard controls now follow the original `JSTX`/`JSTY` idea from `BUMP2`/`REDU2`: key presses accelerate a turn rate and releasing keys damps it back toward centre. Roll ramps faster and higher than pitch, matching the original's finer pitch control; yaw uses the same browser-era rate model even though BBC Elite did not have yaw.
- **Wire mode is hidden-line, not literal wireframe** (`faceByNormalIndex`, `collectWireFaceVisibility`, `wireEdgeVisible`, `drawWireOccluderFaces`) â€” mirrors the original `LL9` strategy: each edge knows its two adjacent faces, face visibility is computed first, then an edge is drawn only if at least one attached face is visible. Visible hull faces are also filled black before stroking the wire edges, so nearer wireframe objects mask farther objects/stars while staying visually "wire". Applies to ships and stations; models without face metadata still fall back to drawing all edges.
- **Armada spawn distance**: the formation's actual spawn position is `local.z` in `spawnArmada` (currently `3800 + ring*340 + (slot%2)*80`, i.e. ~5x the original 760 base) â€” note this is a *different* variable from `center` (also used, for the orbit-target/initial-velocity-direction math), so changing one without the other will desync spawn position from orbit behavior. The general ship-culling distance in `update()` was bumped from 3800 to 6500 to accommodate â€” it was previously despawning armada ships within the same frame they spawned, since 3800 was both the cull radius and roughly the new spawn distance.
- **Ship HUD hp/shield bar** â€” `drawShipLabels` now draws a small colour-graded bar (green/amber/red by remaining hp fraction) under each ship's name/status text, in addition to the existing text.
- **Local & galaxy star charts** (`renderGalaxyMap`, `renderLocalMap`, `game.mapMode`) â€” the Chart panel is inline SVG rather than a text table, like the original's two chart screens. Galaxy map plots every system in the current galaxy at its seeded `(x,y)`; local map is centered on the current system with distance rings (a dashed one at the 7 LY jump range) and only labels systems within ~8 LY by name â€” Lave alone has 100+ neighbours inside 24 LY, so labelling everything in range is unreadable. Clicking a `.map-dot` sets `game.target`, replacing the old per-row Select buttons.
- **Biome-driven planet surface** (`planetFeatureStyle`, `biome` on the planet object from `buildSystemBodies`) â€” economy now picks one of three surface treatments, not just a hue: `economy <= 2` (heavy industrial) â†’ dark base with glowing city-light features that only appear on the shadowed side (`sunFacing < .3`); `economy === 3` â†’ sandy desert with elongated mountain-ridge streaks; `economy >= 4` (agricultural) â†’ ocean base with green/brown continent blobs and white cloud wisps. A dedicated night-side shadow gradient (opposite the actual sun direction, drawn under the features so city lights still glow through it) gives a pronounced day/night terminator, distinct from the older subtle offset-gradient lighting.
- **Ship scale now follows the original source stats** (`SHIP_STATS`, `shipStats`) â€” cross-checking our `MODELS` vertex data against the source confirmed the meshes themselves use the original coordinate units, while the previous collision/scale table was hand-guessed and compressed the size range. `SHIP_STATS.r` is now the source targetable-area radius, `hp` is max energy, and `speed` is source max speed normalized to Cobra Mk III = 1.0; `shipStats()` only falls back to `modelBoundingRadius()` for models without source stats.
- **Combat-orbit direction no longer flips** (`updateShipAI`'s mid-range band, `o.orbitSign`) â€” the orbiting-at-range steering used to scale the entire tangential direction by `jink` (a `Math.sin(...)` term peaking at +-130, i.e. *larger* than the 30-unit inward-pull term it was added to), so every time the sine crossed zero (~every 2.8s) the ship's whole intended heading reversed instead of continuing to circle the same way â€” visible as ships flipping direction and barely moving before flipping again, rather than streaking around in a smooth orbit. Fixed by giving each ship a fixed `orbitSign` (chosen once, lazily) that sets a constant rotational direction, with `jink` now only breathing the orbit radius in and out. Verified against the live game loop: velocity magnitude now holds steady (~65-68) throughout instead of collapsing toward zero at each flip, and the velocity vector sweeps smoothly through a full rotation instead of reversing.
- **Object and station geometry comes from the OBJ/source pass** (`MODELS` before `SHIP_STATS`) â€” missile, escape pod, alloy plate, asteroid, boulder, splinter, rock hermit, Coriolis, and Dodo now use the OBJ folder's vertex/line data with BBC source edge-face/normal metadata where needed so solid rendering can cull and shade faces. The OBJ files contain `v`/`l` data only, so source face metadata is still required for shaded surfaces; the alloy plate remains a special two-sided quad because the original source treats it as a one-face object.
- **Missiles are real tracked projectiles, not hit-scan** (`spawnMissile`, `updateMissile`, `detonateMissile`) â€” both player (`fireMissile`) and NPC (`fireAtPlayer`'s missile branch) firing spawn an actual `game.objects` entry with `type: "missile"` that homes via simple pursuit, orients its nose with the existing `orientShipToVelocity`, and detonates on proximity or fuse expiry. ECM still jams NPC missiles at the same 75% rate as before, just applied at spawn/detonation instead of at instant-hit time.
- **Escape pods, alloy plates, and destructible debris** (`destroyObject`, `spawnEscapePod`, `scoopCargo`) â€” on a player kill there's a 20% chance the pilot ejects (`spawnEscapePod`) instead of the ship simply exploding: this is strictly better for the target (they survive) and worse for the player (no bounty), so it's a real tradeoff rather than a reskin. Non-eject kills have a 50% chance to drop debris, now split between the existing cargo canister and a new alloy plate (`type: "plate"`) â€” both are scoopable via `scoopCargo`, which now also handles the plate's fixed "Alloys" cargo type vs. canisters' random good.
- **Asteroid/boulder/splinter mining loop + Mining Laser** (`spawnAsteroid`, `spawnSplinter`, `spawnMineralCanister`, `EQUIPMENT.miningLaser`) â€” asteroids are the mineable rock type: shooting one apart (`destroyObject`'s `"asteroid"` branch) spawns 2-3 splinters plus a chance of an immediate mineral canister, and splinters themselves have a further chance to drop a mineral canister when destroyed. Boulders are deliberately inert â€” same rock-tumbling physics and laser-targetable, but no reward on breakup â€” a hazard/filler, not a resource. Mineral canisters use a `forcedGood` field (Minerals/Gold/Platinum/Gem-Stones) so `scoopCargo` gives a specific mining payout instead of a random good. The Mining Laser equipment doubles laser damage specifically against rock-type targets (`fireLaser`'s `isRockTarget` check), making dedicated mining meaningfully faster without buffing anti-ship combat.
- **Rock Hermit as a second dockable location** (`dockableTarget`, `type: "hermit"`) â€” `dockableTarget()` generalizes docking to the nearest `station`-or-`hermit` object, used by both `dock()` and `updateAutoDock()`; `game.dockedAt` (set on successful dock) drives the panel subtitle and the docked-view backdrop text instead of both being hardcoded to `${system} STATION`. Unlike the station, a hermit doesn't require slot-rotation alignment to dock (`dock()`'s `slotOk` check is skipped for non-`station` types) since it's a static rock outpost, not a spinning bay. Presence is deterministic per system (`mulberry(hash32(sys.name + ":hermit")) < .3`, ~30% of systems), matching the existing pattern of seeding system-level dressing from the system itself rather than per-visit RNG.
- **Dodo station variant** (`resetFlightScene`'s `isDodo` check, `drawStation`) â€” Anarchy/Feudal systems (`sys.government < 2`, the lawless/frontier end of `GOVS`) spawn a hexagonal-prism Dodo station (`MODELS.dodoStation`) instead of the octagonal Coriolis; `drawStation` now picks the blueprint from `o.model` instead of hardcoding `STATION_MODEL`. Cosmetic/structural only â€” docking mechanics, market, and no-fire-zone logic are unchanged since they already key off `type: "station"`, not the specific model.
- **Elite logo flourish** â€” a small original inline-SVG wordmark (corner-bracket "scope" frame around "VECTOR ELITE" text, using the existing CRT palette vars) at the top of the Help panel (`renderHelp`). Deliberately not a copy of the original diamond logo's design â€” see IP note below.
- **IP scope note (binding for future sessions, not just this one)**: do not copy trademarked artwork (e.g. the Elite diamond logo) even for a personal, unshared project. For geometry/stat work, prefer user-provided/imported data files in the repo and clearly label source-derived tables so future edits know what is factual data versus newly authored gameplay.
- **Other objects no longer visually rotate with the player's own turning** (`worldDirToCam`, used by `drawWireModel`, `drawStation`, `collectSolidDetails`) â€” an object's `rot`/`pitch`/`roll` is a world-frame orientation, but rendering combines its rotated model verts with a camera-*space* position (`cam`, from `cameraTransform`). Combining a world-frame vector with a camera-space position directly is invalid whenever `camera.q` isn't identity: every other ship/station's rendered attitude stayed screen-locked instead of world-fixed, so as soon as the player rolled/pitched/yawed, every other object visibly spun in lockstep with them instead of staying fixed in space. `worldDirToCam(dir)` rotates a world-frame direction by `quatConj(camera.q)` before it's combined with `cam`; this is the same correction `cameraTransform` already applies to object *positions*, just also applied to the rotated-model-vertex vector. Verified numerically (a world-frame direction fed through `worldDirToCam` exactly matches the ground-truth `cameraTransform(pos + dir) - cameraTransform(pos)`) and visually (a fixed-orientation ship's on-screen bounding box rotates 90Â° to match a 90Â° player camera roll, instead of staying put).
- **Fixed several ships missing whole faces/edges** (`buildBlueprint`) â€” face reconstruction was gating which edges count toward a face's vertex set on `edgeVisibility` (`< model.hullVisibility || 29`), a per-edge 5-bit viewing-quadrant bitmask carried over from the BBC source's real-time hidden-line performance hack. That's a fundamentally different concept from face *topology* (which edges bound which face) and has nothing to do with it â€” it happened to zero out most of an edge set for several models (ferdelance was worst-hit: only 1 of 10 faces ever got built), leaving gaping holes in the solid-shaded hull and sparse, disconnected wireframes with most edges missing. `edgeVisibility` is otherwise unused now that hidden-line visibility is computed per-frame from actual face-normal facing (`collectWireFaceVisibility`/`wireEdgeVisible`, see above), so the filter was simply removed from `buildBlueprint`; every edge listed in `edgeFaces` for a given face now always contributes to that face. Verified by asserting `faces.length === normals.length` for every entry in `MODELS` (previously failed for asp, boa, cobra1, ferdelance, moray, shuttle, transporter) and visually confirming complete, gap-free hulls in both wire and solid mode.
- **Ship/station wire and solid rendering share one face-fill engine, plus a third "both" graphics mode** (`collectSolidFaces`'s new `flat` param, `faceFacing`, `game.graphicsMode` now `"wire" | "solid" | "both"`) â€” wireframe mode used to fill camera-facing faces black via a separate bespoke pass (`collectWireOccluderFaces`/`drawWireOccluderFaces`, now deleted) purely to occlude what's behind, duplicating the face-visibility/projection logic `collectSolidFaces` already had for lit shading. `collectSolidFaces(model, camVerts, points, baseColor, alpha, flat)` now produces either a flat black occluder fill (`flat: true`, used by wire mode) or the full lit/textured fill (`flat: false`, solid/both) from the same face loop, both fed through the same `drawSolidItems`; the face-facing test itself (`faceFacing`) is also shared with `collectWireFaceVisibility` rather than being computed twice. `drawWireModel`/`drawStation` now read as `mode === "wire" ? flat fill : lit fill`, then `mode !== "solid"` strokes hidden-line edges on top â€” so "both" is just "solid fill" + "wire edges" falling out of the same two conditions, not a separate code path. These three had a genuine two-different-face pair instead (a data-entry inconsistency, not an algorithmic gap), so once the `edgeVisibility` filter above was removed, these spike-tip vertices got pulled into whatever real hull face they bordered, producing a thin triangular sliver connecting the spike back to the hull. Fixed by correcting those three `edgeFaces` entries to self-referencing pairs, matching the convention already used everywhere else. Checked every `details` entry (`line`/`polyline`/`engine`) across all 31 `MODELS` for edges with mismatched `edgeFaces`, confirmed only these three, and reverified `faces.length === normals.length` still holds everywhere after the fix (i.e. this didn't reintroduce the missing-face bug from before).

- **Ambient ships colliding with the station** â€” the station's declared `r` (160/180, used nowhere for spatial reasoning) badly undersold its real rendered size (`modelBoundingRadius`: ~226 Coriolis, ~247 Dodo); nothing in ambient-traffic spawn/AI code accounted for the station's actual bulk at all. Cull distance bumped 6500 -> 12000 to match. Checked bbcelite.com's space station safe zone deep dive for comparison; its distance constants are in the BBC source's own fixed-point unit system tied to planet radius, not directly convertible to this engine's units, so the fix here is grounded in this codebase's own actual object sizes rather than a ported constant.

- **Ship hulls can round-trip through a real 3D editor via `ship_models/*.obj`** (`modelToObjText`, `exportShipObj`, `parseObjText`, `buildBlueprintFromFaces`, `loadShipModelsFromObj`) â€” rather than reporting hull problems (features modeled but never visible, back-face-culled away, etc.) ship by ship for hand-editing of the `edgeFaces` adjacency data in JS, any ship's current mesh can be exported as a real Wavefront `.obj` ("Export OBJ" button in the Ship panel, or `exportAllShipObjs()` from the console) and edited in Blender or similar. Import derives edges/adjacency/normals/UV *from the OBJ's actual faces* (`buildBlueprintFromFaces`) rather than reconstructing faces from hand-authored edge data like `buildBlueprint` does â€” there's no reconstruction step to get wrong, so this whole path is immune to the missing-face/wrong-adjacency bug class fixed earlier in this file.
